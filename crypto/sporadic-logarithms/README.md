# Sporadic Logarithms

**Category:** Crypto | **Difficulty:** Medium-Hard | **Flag:** `bctf{th3_m0n5t3r_gr0up_w45_t0_1mpr4ct1c4l_:(}`

## TL;DR

A "discrete logarithm" problem set in the holomorph of GL(3, 65537) — invertible 3x3 matrices over a finite field. The trick is that the conjugator `c` has small order (at most 8), which lets you decompose the holomorph exponentiation as `G^q * P_r` and solve it with Baby-step Giant-step (BSGS) using only ~750 oracle queries out of the allowed 10000.

## What We're Given

We get `chall.py`, the full challenge server source. It runs 5 rounds; pass all 5 and you get the flag.

Each round:
- We're working in **GL(3, 65537)** — that's the group of all invertible 3x3 matrices whose entries live in GF(65537), the finite field with 65537 elements.
- The server picks a random element `g` from this group and a special "conjugator" matrix `c` with small order (a divisor of 65536, and at most 8).
- It computes `h = hol_pow((g, c), x_secret)` for a secret `x_secret` in [0, 262144].
- Our job: find any `x` in [0, 262144] that produces the same `h`. We get up to 10000 oracle queries per round to help us, where we can ask for matrix products (`mul`), inverses (`inv`), and conjugation by `c` (`phi(a) = c * a * c^-1`).

The flag title references the **Monster group** — the largest of the 26 "sporadic" finite simple groups, with about 8 × 10^53 elements. The flag text `th3_m0n5t3r_gr0up_w45_t0_1mpr4ct1c4l` gives away that they originally wanted to use it but couldn't. GL(3, 65537) is much more tractable — which is exactly what we exploit.

## Initial Recon

The first thing that stands out in the source is `hol_pow`:

```python
def hol_mul(x, y):
    a, c1 = x
    b, c2 = y
    return a * (c1 * b * c1.inverse()), c1 * c2

def hol_pow(base, e, F, n: int):
    one = identity_matrix(F, n)
    result = (one, one)
    cur = base
    k = e
    while k > 0:
        if k & 1:
            result = hol_mul(result, cur)
        cur = hol_mul(cur, cur)
        k >>= 1
    return result
```

This is **holomorph multiplication** — a construction where you take a group G and an automorphism phi, and the "holomorph" is all pairs `(g, phi^k)` with the product rule `(a, phi^i) * (b, phi^j) = (a * phi^i(b), phi^(i+j))`.

In our case, `phi` is conjugation by `c`: `phi(a) = c * a * c^-1`. So `hol_pow((g, c), x)` computes a product of `x` copies of `(g, c)` using this rule. The key question: what does `hol_pow((g, c), x)[0]` actually look like as a matrix?

Expanding the product rule by hand:

```
(g, c)^1 = (g, c)
(g, c)^2 = (g * c*g*c^-1, c^2)
(g, c)^3 = (g * c*g*c^-1 * c^2*g*c^-2, c^3)
...
(g, c)^x first component = g * phi(g) * phi^2(g) * ... * phi^(x-1)(g)
```

Which means:

```
hol_pow((g, c), x)[0] = prod_{i=0}^{x-1} c^i * g * c^{-i}
```

We're computing a product of `x` conjugates of `g` by successive powers of `c`.

## The Vulnerability / Trick

Here's where the small order of `c` becomes a weapon.

`c` has order `m` (at most 8), which means `c^m = identity`. So the conjugates cycle: `c^(m+i) * g * c^{-(m+i)} = c^i * g * c^{-i}`. After `m` steps, you're back to the same conjugate you started with.

This means we can write `x = q*m + r` (where `0 <= r < m`) and split the big product into full cycles plus a partial cycle:

```
h = (g_0 * g_1 * ... * g_{m-1})^q * (g_0 * g_1 * ... * g_{r-1})
  = G^q * P_r
```

where:
- `g_i = c^i * g * c^{-i}` — the i-th conjugate (repeating with period m)
- `G = g_0 * g_1 * ... * g_{m-1}` — product over one full period
- `P_r = g_0 * g_1 * ... * g_{r-1}` — partial product for remainder r (with `P_0 = identity`)

So instead of a DLP in a huge group over [0, 262144], we now have:

> Find `q` such that `G^q = h * P_r^{-1}` (for some `r` in 0..m-1)

This is a standard discrete logarithm problem in the cyclic subgroup generated by `G`. And since `x <= 262144` and `m <= 8`, we have `q <= 262144/2 = 131072` at most. That's a search space of sqrt(131072) ≈ 362 in each direction for BSGS. Totally feasible.

**Baby-step Giant-step (BSGS)** is the classic DLP speedup: instead of trying all `q` values one by one, you precompute a lookup table of `G^j` for j = 0..step-1 ("baby steps"), then for each "giant step" i you check if `target * (G^step)^{-i}` lands in that table. If `target = G^q` and `q = i*step + j`, you'll find the match. Time and space O(sqrt(q_max)) instead of O(q_max).

This is a textbook BSGS, but the interesting twist here is that the "group elements" are matrices you can only access through oracle handles — you never see the actual matrix values, just integer handles. The oracle still gives you `mul`, `inv`, and `eq` (implicitly, through `phi`), which is all BSGS needs.

## Building the Exploit

Here's the plan, step by step:

**Step 1: Compute all the conjugates `g_i`**

We already have `g` (handle to g_0). Apply `phi` repeatedly to get g_1, g_2, ..., g_{m-1}:

```python
gi = [g_h]
cur = g_h
for i in range(1, mo):
    cur = bb.phi(cur)
    gi.append(cur)
```

This costs `m-1` oracle queries (at most 7).

**Step 2: Build the period product G and partial products P_r**

Multiply them up one at a time. We need all `P[r]` values (for r = 0..m) where `P[m] = G`:

```python
P = [one_h]   # P[0] = identity
cur = one_h
for i in range(mo):
    cur = bb.mul(cur, gi[i])
    P.append(cur)
G = P[mo]     # P[m] = G, the full period product
```

This costs `m` oracle queries. Now `G` is the "generator" we'll do BSGS over.

**Step 3: Precompute baby steps**

```python
max_q = BOUND // mo + 1   # max possible value of q
step = int(math.isqrt(max_q)) + 1

baby = {}
Gpow = one_h
for j in range(step):
    if Gpow not in baby:
        baby[Gpow] = j
    if j < step - 1:
        Gpow = bb.mul(Gpow, G)
G_step = bb.mul(Gpow, G)        # G^step
G_step_inv = bb.inv(G_step)     # (G^step)^{-1}
```

The baby step table maps "handle of G^j" to j. This costs about `step + 2` queries.

Note: we're using oracle handles as dictionary keys. Since the server assigns the same handle to the same matrix (it checks `mat_key` and deduplicates), equal matrices get equal handles — so handle equality implies matrix equality. That's what makes this work without ever seeing the actual matrix entries.

**Step 4: Giant steps, trying each r**

For each possible remainder r:

```python
for r in range(mo):
    Pr_inv = bb.inv(P[r])
    target = bb.mul(h_h, Pr_inv)   # target = h * P_r^{-1} = G^q (if r is correct)
    
    cur_t = target
    for i in range(max_q // step + 2):
        if cur_t in baby:        # cur_t = G^q * (G^step)^{-i} = G^{q - i*step}
            j = baby[cur_t]
            q = i * step + j
            x = q * mo + r
            if 0 <= x <= BOUND:
                ok = bb.submit(x)
                return ok
        cur_t = bb.mul(cur_t, G_step_inv)
```

For each r, we compute the "target" value `h * P_r^{-1}`. If we guessed r correctly, this equals `G^q`. Then we do the giant step walk: multiply by `(G^step)^{-1}` each iteration. When `cur_t = G^{q - i*step}` lands in our baby table, we know `j = q - i*step`, so `q = i*step + j`.

Total oracle queries per round: `(m-1) + m + (step+2) + m*(2 + step)` ≈ `8 + 8 + 365 + 8*367` ≈ 3317 for m=8, well under the 10000 limit.

## Key Takeaways

**The core trick: periodicity kills hard problems.** The holomorph exponentiation looks intimidating at first, but the small order constraint on `c` means the conjugates `g_i` repeat. Once you spot that, a scary-looking "sporadic logarithm" falls apart into a standard BSGS over a manageable range.

**Decompose, then conquer.** Writing `x = q*m + r` is a classic factoring move in group theory. Any time you have a periodic structure, look for a way to separate the "how many full cycles" (q) from the "leftover" (r). The full cycle product G becomes your new generator, and m small partial products cover all the remainders.

**Black-box groups with handle oracles.** This challenge introduces an interesting setup where you never see the group elements — only handles. BSGS still works because equality of handles implies equality of matrices (the server deduplicates), and that's the only equality check BSGS needs. The algorithm is purely "is this element in my baby table?" and "multiply by this inverse." No matrix arithmetic on our side at all.

**Why the Monster Group was "too impractical."** The flag text is a confession: the challenge author probably wanted to use an actual sporadic group (maybe even the Monster, which has order ~8 × 10^53). The problem is that just *representing* Monster Group elements requires around 196882-dimensional matrices, and doing even one multiplication would be absurdly expensive. GL(3, 65537) is the tractable substitute — and the vulnerability (small-order conjugator) still applies regardless of the group.
